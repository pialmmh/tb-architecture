# apibuilder — versioned API-contract authoring skill

Use this skill whenever the task is to **add, change, or version-bump documentation** for a service's API contract under `architecture/docs/api-doc/<service>/`. The skill is also valid as a standalone reading reference — point another agent at this file and they will know the conventions without any extra context.

> **Canonical live example:** `/home/mustafa/telcobright-projects/architecture/docs/api-doc/party/` — read it before authoring; copy its shape, don't invent a new one.

---

## 1. Invocation

```
/apibuilder <version> <prompt>
```

- `<version>` — the target version directory the work goes into. Forms accepted:
  - `v1` (resolves to the highest `v1.X/` present, e.g. `v1.0`)
  - `v1.0`, `v1.1`, `v2.0`, … (exact)
  - `latest` (resolves to the highest version present)
  - `next` (start a new version dir; see §3 for when to do this)
- `<prompt>` — natural-language description of what to add/change. The skill maps the prompt onto the correct input/trigger/output node(s).

**Examples**
- `/apibuilder v1 "add an access control list for keycloak client"`
- `/apibuilder v1.0 "expose a partner-search input that returns a partner list"`
- `/apibuilder next "breaking: createTenant now requires regionCode"`

If the user just wants the conventions and is not invoking the skill, treat this file as **read-only reference docs** — explain, don't generate.

## 2. The model — input → trigger → output

Three sibling top-level dirs inside each version folder:

```
<service>/
└── vMAJOR.MINOR/
    ├── input/      what the service receives (api call, kafka msg, clock tick, config load, …)
    ├── trigger/    named business event/action; fires when an input arrives, OR on a clock,
    │               OR as a chain reaction from another trigger
    └── output/     named, typed artifact produced by a trigger
                    (json, db row, kafka msg, file, log)
```

```
input  ─fires→  trigger  ─produces→  output
                   ↑                    ↑
              (also fired by         (output may have
               clock or another       multiple triggers)
               trigger)
```

Why three layers and not two (just input → output)? Because real services have outputs that are not produced by an HTTP call — they're produced by a clock tick, by another trigger firing, by a Kafka event. A "monthly recurring invoice" is generated by a billing-cycle trigger that fires on the clock; the original `createSubscription()` input only **arms** that trigger. Two-layer models can't express this without lying.

Per-file rules:
- **input/** — describe the call/event shape and **list which trigger(s) it fires**. Don't list outputs directly.
- **trigger/** — describe activation source(s), what the trigger does, and **list inputs(s) and output(s)**. Triggers carry the idempotency/retry contract.
- **output/** — `name`, `type`, `trigger(s)`, brief description. Outputs are passive artifacts, never doers.

## 3. Versioning

`vMAJOR.MINOR/` directories. Each is a self-contained snapshot.

| Change | Bump |
|---|---|
| New input / trigger / output | MINOR |
| New optional field on existing one | MINOR |
| Removed input / renamed trigger / changed output shape | MAJOR |
| Added required field to existing input | MAJOR |
| Typo, prose clarification | no bump |

Rules:
- **A version dir freezes once a release using it ships.** Edits after that are limited to clarifications that don't change the wire contract.
- To start a new version, **copy the latest version dir wholesale** to the new name and edit there. Don't delta-edit; the new version must be self-contained.
- Cross-links **stay within the same version**. If `v1.1` reuses a `v1.0` contract verbatim, copy the file across. Never link `v1.1/...` → `v1.0/...`.
- The "current" version is whichever folder name is highest. No symlinks.
- `next` in the invocation means "start a new version that succeeds the current top one". Decide MAJOR vs MINOR from the prompt's nature.

## 4. Filename + naming conventions

- **camelCase** action names for inputs, triggers, outputs.
  - input: `createTenant.md`, `kcValidateCredentials.md`, `subscriptionRenewedTick.md`
  - trigger: `tenantProvisioning.md`, `provideAuthToKeyCloak.md`, `monthlyBillingCycle.md`
  - output: `tenant.md`, `dbcreated.md`, `kcUserView.md`, `recurringInvoice.md`
- **Long, self-descriptive names beat short cryptic ones.** A reader skimming the directory listing should infer the responsibility from the filename alone.
- Inputs grouped by kind: `input/api/`, `input/kafka/`, `input/clock/`, `input/config/`. Add new kinds as needed; keep `input/api/` flat.
- Triggers and outputs sit flat in their dirs (no sub-grouping) until there are >20 of either, then group by area.

## 5. File templates

### 5.1 input/api/<verb><Noun>.md

```markdown
# <verbNoun>()

HTTP API call. <one-sentence purpose>.

<optional 1–2 line context — security/auth, idempotency caveats, who calls this>.

## Params

- `<name>` (<Type>, required|optional) — <description>
- ...

## Computed (not supplied by caller)

- <field> — <how it's derived; only include if non-trivial>

## Fires trigger

- [trigger/<name>](../../trigger/<name>.md)
```

For non-API inputs (kafka, clock, config, env), swap the first heading to e.g. `# subscriptionRenewalTick (cron)` and document the activation cadence + payload.

### 5.2 trigger/<name>.md

```markdown
# <triggerName>

<One-sentence business meaning of this trigger.>

## Activated by

- input: [input/<kind>/<name>](../input/<kind>/<name>.md)
- (or) clock: <cron expr or cadence>
- (or) chained from: [trigger/<other>](./<other>.md)

## What it does

<3–8 line description of the steps the trigger runs, in order.>

## Produces

- [output/<name>](../output/<name>.md)
- ...

## Idempotency & retries

<How re-running the trigger behaves. Any deduping key. Whether earlier steps roll back on later-step failure.>

## Notes

<Production caveats, security, dev-vs-prod differences.>
```

### 5.3 output/<name>.md

```markdown
name: <name>
type: <Shape> (json | db row | kafka msg | file | log line | composite)
trigger: [<triggerName>](../trigger/<triggerName>.md)  // list multiple if multiple

<2–4 line description of the artifact, including which fields are notable, any state machine, any never-returned fields.>
```

Outputs are intentionally compact — name/type/trigger lines are the contract; the prose is a reading aid.

## 6. Cross-linking rules

- Every input file lists at least one **trigger** it fires.
- Every trigger file lists at least one **activation source** (input or clock or another trigger) **and** at least one **output** it produces.
- Every output file lists at least one **trigger** that produced it.
- Use **relative markdown links** with the `../` form, never absolute filesystem paths.
- Never link across versions. If a v1.0 file is identical in v1.1, copy it.

## 7. Workflow when the skill is invoked

Given `<version>` and `<prompt>`:

1. **Resolve the service.** Default = the service whose api-doc folder the user is currently working in (cwd hint, last-edited dir). If ambiguous, ask once.
2. **Resolve the version dir.**
   - `v1` / `latest` → highest existing dir matching pattern.
   - `v1.0`, etc. → exact, error if missing.
   - `next` → bump from current top per §3 rules; create dir by copying latest.
3. **Read the version's existing nodes.** List `input/api/*`, `trigger/*`, `output/*` so you understand what already exists. Avoid duplicate names; reuse existing triggers when the new behaviour fits.
4. **Decide which nodes the prompt needs.**
   - A new endpoint usually = one input + one trigger + one or more outputs.
   - A new field on an existing input = edit the input file (and bump if required-field).
   - A new clock-driven artifact = one trigger (clock-activated) + one or more outputs.
   - A new federation/integration = often one input + one trigger + one output, with security notes in the trigger.
5. **Write or edit files** following §5 templates.
6. **Wire cross-links** per §6.
7. **Decide if a version bump is required** per §3. If MAJOR, you should have used `next` — if not, surface the contradiction and pause.
8. **Surface unanswered questions** (security boundary, idempotency contract, who's the caller) — don't invent.

## 8. Worked example

Invocation: `/apibuilder v1 "add an access control list for keycloak client"`

Interpretation:
- Service: `party` (Keycloak federation lives there).
- Version: `v1` → resolves to `v1.0`.
- Conceptual model: Keycloak admin needs to set/update a per-client ACL stored in Party. Party serves it back when Keycloak asks.
- Two new inputs (set + read), one trigger ("kcClientAclManagement"), two outputs ("kcClientAcl" json, "aclStored" event).

Files created under `v1.0/`:

```
input/api/
  setKcClientAcl.md            ← POST set/replace ACL for a (realm, clientId) pair
  getKcClientAcl.md            ← GET ACL — read path the SPI calls during token issuance
trigger/
  kcClientAclManagement.md     ← activated by both inputs; produces both outputs
output/
  kcClientAcl.md               ← json shape returned on read
  aclStored.md                 ← side-effect: master DB row written + audit row
```

Each file follows §5 templates. Cross-links wired per §6. No version bump (additive only).

## 9. When not to use this skill

- **Per-endpoint reference docs.** Those live under `<service>/todo/` (the parked OpenAPI-ish style) and are out of scope here. The skill operates only on the input/trigger/output graph.
- **Code generation.** This skill writes documentation, not Java. Code work needs a different skill.
- **Cross-service contracts** that don't fit one service's input/trigger/output. Those go in `architecture/docs/<topic>/` directly.
